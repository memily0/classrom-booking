### 2. Аутентификация

Бэкенд:

Задача: создать систему авторизации и написать для нее базовые тесты. Иницализировать таблицу Users в базе данных - поля. Должна быть возможность проверки корректности пароля, данных и существования пользователя

Таблица:

| Поле         | Тип       | Пояснение                                                  |
| ------------ | --------- | ---------------------------------------------------------- |
| id           | Long      | Уникальный id пользователя в базе данных                   |
| email        | String    | Почта пользователя, по которой он регистрируется           |
| passwordHash | String    | Хэшированный пароль пользователя, с которым идет сравнение |
| name         | String    | Имя человека в системе                                     |
| surname      | String    | Фамилия                                                    |
| createdAt    | Timestamp | Время создания пользователя в системе                      
Требования к таблице  
- `email` должен быть уникальным 
- Пароль нельзя хранить в открытом виде  
- В базе хранится только passwordHash
- При инициализации базы создать несколько технических аккаунтов для тестирования

Критерии приемки: таблица иницализирована, кидаются корректные http коды при ошибках, данные пароля хэшируются и обрабатываются.
Изначально создаем несколько технических аккаунтов для тестирования

Фронтенд: 

Задача: создать страницу аутентификации. Пользователь вводит данные, система отправляет их на backend - ошибки корректно парсятся и отображаются. После входа данные отображаются я на странице в виде иконки - пользователь может увидеть их (либо отдельная страницу, либо dropdown окно).
Также нужна кнопка выхода
Логика работы:
1. Пользователь открывает страницу `/login`.
2. Вводит email и пароль.
3. Frontend отправляет запрос:

```
POST /api/auth/login
```

4. Если ответ успешный:
    - сохранить `accessToken`;
    - сохранить данные пользователя;
    - перенаправить пользователя на главную страницу;
    - отобразить иконку пользователя.
5. Если ответ с ошибкой:
    - показать сообщение об ошибке;
    - не авторизовывать пользователя.

Критерии приемки: создана страница для входа, на нее можно перейти с главной страницы, при входе данные отображаются в dropdown от иконки либо отдельная страница, ошибки показываются

## 3. Список кабинетов

### Цель  
  
На главной странице пользователь должен видеть список кабинетов и иметь возможность выбрать дату/время, на которое ему нужен кабинет. После выбора времени система показывает, какие кабинеты свободны, какие заняты, и на какой период доступен каждый кабинет

## 3.1. Получение доступности кабинетов на выбранное время

Бэкенд:

### Цель

Нужно реализовать backend-логику, которая по выбранным пользователем дате и времени возвращает список кабинетов с информацией о том, свободен кабинет или занят в этот момент.

Важно: занятость не нужно хранить отдельными слотами по часам. Нужно хранить бронирования как интервалы времени, а доступность кабинетов вычислять на основе пересечения выбранного времени с существующими бронированиями.
## Таблицы

### rooms

Таблица кабинетов.

| Поле      | Тип     | Описание                     |
| --------- | ------- | ---------------------------- |
| id        | Long    | Уникальный id кабинета       |
| number    | string  | Номер кабинета               |
| is_active | boolean | Можно ли бронировать кабинет |
### bookings

Таблица бронирований.

| Поле       | Тип         | Описание                           |
| ---------- | ----------- | ---------------------------------- |
| id         | Long        | Уникальный id брони                |
| room_id    | Long        | Кабинет                            |
| user_id    | Long        | Пользователь, который создал бронь |
| start_time | timestamp   | Начало брони                       |
| end_time   | timestamp   | Конец брони                        |
| status     | enum/string | Статус брони                       |
| created_at | timestamp   | Дата создания                      |
| updated_at | timestamp   | Дата обновления                    |

Возможные статусы брони:

active - активная бронь
cancelled  - отмененная бронь
available - доступен

Эндпоинт:

GET /api/rooms/availability?datetime=2026-05-21T14:00:00

Query параметры:

|Параметр|Тип|Обязательный|Описание|
|---|---|---|---|
|datetime|ISO datetime|да|Момент времени, на который пользователь ищет кабинет|
Правило занятости в конкретный момент:

Кабинет считается занятым в выбранный момент, если существует активная бронь:

```sql
booking.start_time <= selected_datetime
AND booking.end_time > selected_datetime
AND booking.status = 'active'
```

Ответ

```json
[
  {
    "id": "room_1",
    "number": "305",
    "isAvailable": true,
    "status": "available",
    "availableUntil": "2026-05-21T16:30:00",
    "busyUntil": null
  },
  {
    "id": "room_2",
    "number": "204",
    "isAvailable": false,
    "status": "busy",
    "availableUntil": "2026-05-21T16:30:00",
    "busyUntil": "2026-05-21T15:00:00"
  }
]
```

## Дополнительно: availableUntil

Желательно также возвращать `availableUntil` для свободных кабинетов.
Это время ближайшей будущей брони этого кабинета после выбранного времени.

Например:
Пользователь выбрал 14:00
Кабинет 305 свободен сейчас, но у него есть бронь с 16:30 до 18:00
Значит:status = available
availableUntil = 16:30

На главной странице пользователь должен выбрать дату и время, после чего увидеть список кабинетов с актуальным статусом доступности на выбранный момент.

### Карточка кабинета

### Поля карточки

| Поле           | Обязательное | Описание                        |
| -------------- | ------------ | ------------------------------- |
| Номер кабинета | да           | Например: 305                   |
| Статус         | да           | Отмапить статусы с бэкенд-таски |
| Время          | желательно   | Свободен до "время"             |

### Критерии приемки

1. До запроса показываются все кабинеты в текущий момент, то есть в момент загрузки страницы
2. Во время запроса показывается loading состояние
3. После неудачного ответа корректно отображаются ошибки
4. Карточки отсортированы, свободные - выше, занятые - внизу

4. Если интервал свободен — создается бронь.
5. Если интервал уже занят — пользователь получает ошибку.

# Backend

## Что нужно сделать

Backend должен реализовать два endpoint:

1. Получение расписания кабинета на дату.
2. Создание бронирования.

# Endpoint 1: расписание кабинета

```
GET /api/rooms/{roomId}/schedule?date=2026-05-21
```

## Что делает

Backend должен:

1. Проверить, что кабинет существует.
2. Получить рабочие часы на выбранную дату.
3. Получить все активные бронирования кабинета на эту дату.
4. Отсортировать брони по времени начала.
5. Посчитать свободные интервалы между бронированиями.
6. Вернуть:
    - информацию о кабинете;
    - рабочие часы;
    - занятые интервалы;
    - свободные интервалы.


## Как считать свободные интервалы

Допустим рабочее время кабинета:

```
08:00–20:00
```

Брони:

```
10:00–11:30
14:00–15:00
```

Тогда свободные интервалы:

```
08:00–10:00
11:30–14:00
15:00–20:00
```

### Endpoint 2: создание бронирования
